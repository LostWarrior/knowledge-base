# Design Document

Architecture and design decisions behind `kb`.

## Research Summary

Before building `kb`, we surveyed existing tools for managing persistent knowledge across coding sessions. Each influenced the design in specific ways:

### Beads

Beads uses structured metadata (schemas, typed fields) to make knowledge entries machine-queryable. We borrowed the idea of **disciplined frontmatter** - every entry has a consistent YAML header that enables both human scanning and programmatic filtering. Where Beads uses JSONL and a custom runtime, we kept the storage as plain markdown.

### ReMe

ReMe demonstrated that **markdown-first storage with compaction** is viable for long-lived knowledge. Their approach of summarizing old entries to reduce context size directly inspired our `kb compact` command and the tiered lifecycle model. Active entries stay verbose; archived entries get compressed.

### Mem0

Mem0 handles **deduplication and memory decay** well - entries that are no longer relevant fade out rather than accumulating forever. Our TTL-based staleness detection borrows this concept but implements it with simple date arithmetic in bash rather than vector similarity scores. Mem0's dependency on external vector stores and APIs made it unsuitable as a local-first tool.

### Cognee

Cognee's approach to **knowledge graph construction** from unstructured text was interesting but overkill for our use case. The key takeaway was that relationships between entries matter. We handle this lightly through domain tags and cross-references in frontmatter rather than building a full graph.

### Letta

Letta (formerly MemGPT) provides sophisticated long-term memory for AI agents with tiered storage. Their **core memory vs archival memory** split directly inspired our active/reference/archive tiers. However, Letta requires running a server process and managing state through an API - far too heavy for a tool that should work in any terminal.

## Why Markdown over JSONL/SQL

**Human readability is non-negotiable.** Engineers need to read, edit, and review knowledge entries in their normal workflow - in their editor, in git diffs, in pull request reviews. Markdown satisfies all of these.

JSONL is machine-friendly but hostile to humans. Opening a JSONL file to review or edit an entry means parsing nested JSON mentally. Git diffs of JSONL are noisy and hard to review.

SQL (SQLite or similar) provides powerful querying but introduces a binary file that breaks git diffing entirely. It also requires tooling to inspect - you can't just `cat` a database.

Markdown with YAML frontmatter gives us the best of both worlds:
- Human-readable and editable with any text editor
- Structured enough for programmatic access via `grep`, `awk`, or `sed`
- Git-friendly with clean, reviewable diffs
- No special tooling required to read or modify

## Why INDEX.md over Grep-First Discovery

The central question: how should an agent (or human) discover what knowledge exists?

**Option A: Grep-first.** Walk the directory, grep for patterns, build a mental model. This works but is expensive. An AI agent doing `find . -name "*.md" | xargs grep "domain:"` might spend 500+ tokens just to discover what entries exist. Every session starts with this overhead.

**Option B: INDEX.md.** A single auto-generated file that catalogs every entry with its ID, title, tier, domain, and last-updated date. One file read, ~150 tokens, complete picture.

We chose Option B as the **primary** discovery mechanism and kept Option A as a power-user escape hatch. The INDEX.md is regenerated by `kb index` and stays accurate as long as users run it after changes (or hook it into git commits).

The tradeoff is that INDEX.md can go stale. We mitigate this with:
- `kb doctor` warns when INDEX.md is out of date
- `kb validate` checks consistency between entries and the index
- Git hooks can auto-run `kb index` on commit

## Why Tiered Directories over Flat + Tags-Only

A flat directory with tags-only organization was considered and rejected for several reasons:

**Cognitive load.** A flat directory with 50+ entries is overwhelming. Tiers provide immediate visual triage - `active/` has 5 files that matter right now, `archive/` has 30 files you can ignore.

**Agent efficiency.** An agent instructed to "read active entries" can scan one directory. With flat + tags, the agent must open every file, parse frontmatter, and filter - multiplying token cost by the total entry count.

**Lifecycle semantics.** The directory a file lives in communicates intent:
- `active/` - "this is current, read it"
- `reference/` - "this is stable, consult when needed"
- `archive/` - "this is historical, ignore unless researching"

Tags still exist for cross-cutting concerns (domain, topic) but the tier is the primary organizational axis.

**Filesystem as API.** `ls active/` is the simplest possible query. No parsing, no tooling, no flags. This matters for both humans and agents.

## Why Bash over Go/Rust/Python

**Zero dependencies.** Bash and standard unix tools (`grep`, `sed`, `awk`, `date`, `find`, `sort`) exist on every developer machine. No `go install`, no `cargo install`, no `pip install`, no `npm install`. No version conflicts, no virtual environments, no build step.

**Universal.** Works on macOS, Linux, WSL, any CI runner, any Docker container. The only requirement is a POSIX-compatible shell.

**Forkable.** Engineers can read the source, understand it in minutes, and modify it for their workflow. A 500-line bash script is more approachable than a 5000-line Go project with 15 source files and a build system.

**Appropriate complexity.** `kb` manipulates text files, generates markdown, and does date arithmetic. These are tasks bash handles well. There's no network I/O, no concurrent processing, no complex data structures that would benefit from a systems language.

The tradeoff is that bash has weaker error handling, no type system, and can be fragile with edge cases (filenames with spaces, unicode). We mitigate this with:
- Strict mode (`set -euo pipefail`) in all scripts
- Quoting all variables
- Input validation and sanitization
- Comprehensive test coverage

## Security Model

`kb` processes user-supplied input (entry titles, IDs, tags) and uses it in file paths and shell operations. The security model is defensive:

**No eval.** We never pass user input to `eval` or execute it as code. All string processing uses parameter expansion, `sed`, or `awk`.

**Path validation.** Entry IDs are sanitized to alphanumeric characters, hyphens, and underscores. Any path traversal attempts (`../`, absolute paths) are rejected before reaching the filesystem.

**Input sanitization.** Titles and tags are stripped of shell metacharacters before being written to frontmatter or used in filenames.

**Least privilege.** `kb` only reads and writes within the vault directory. It never modifies files outside the vault root, never accesses the network, and never requires elevated permissions.

## Dual-Output Indexing

`kb index` generates two outputs from the same entry scan:

- **INDEX.md** - a markdown table for humans browsing GitHub, their editor, or a terminal. Columns include ID, title, tier, domain, and last-updated date.
- **.kb/manifest.json** - a JSON array for AI agents and scripts. Same data, structured for programmatic access.

**Why both?** A 300-row markdown table costs an AI agent roughly 10,000 tokens just to parse. Most of those tokens are wasted - the agent only needs 2-3 entries. With a JSON manifest, an agent can run a targeted `jq` query and retrieve just the relevant subset for around 200 tokens.

```bash
# Agent reads only what it needs
jq '.entries[] | select(.domain == "backend" and .tier == "active")' .kb/manifest.json
```

INDEX.md remains the human-facing view - it renders nicely on GitHub, is easy to scan in an editor, and works without any tooling. manifest.json is the agent-facing view - cheap to query, easy to parse, and structured for filtering.

This dual-output approach was added based on feedback that INDEX.md alone doesn't scale for agent consumption. As vaults grow past 50-100 entries, the token cost of parsing a large markdown table becomes a real bottleneck for AI agent workflows.

## Compact vs Distill

`kb` offers two distinct commands for reducing entry size, separated by a clear design boundary:

**`kb compact`** is deterministic. It concatenates, deduplicates, and shortens entries using pure text processing - no external dependencies, no API keys, no network access. It runs the same way every time on the same input. This is the baseline capability that works everywhere.

**`kb distill`** is LLM-powered. It calls an external language model to produce higher-quality summaries that understand context, preserve nuance, and generate readable prose. It requires an API key and network access.

**Why separate them?** The core design principle of `kb` is zero dependencies. Requiring an API key for basic functionality would violate that. By keeping `compact` as a standalone, deterministic operation and offering `distill` as an opt-in upgrade, the tool remains fully usable without any external services while still offering LLM-powered capabilities for users who want them.

The `--keep` flag on `distill` preserves the original files alongside the summary, so the operation is non-destructive by default.

## Migration Architecture

`kb migrate` imports an existing markdown directory into a structured vault. The design follows a three-phase pipeline inspired by Terraform's plan/apply model.

### Phase 1: Scan

Walk the source directory and classify every file. A parser chain extracts metadata in priority order:

1. **YAML frontmatter** - if present, use title/status/domain directly
2. **Informal headers** - detect `**TL;DR**`, `**Status**`, or similar bold-text conventions
3. **H1 heuristic** - use the first `# Heading` as the title
4. **Filename fallback** - derive title from the filename (strip extension, convert hyphens/underscores to spaces)

The source directory is treated as strictly read-only throughout the entire process.

### Phase 2: Preview

Display a full migration plan showing each file's proposed destination tier, generated title, and any compaction groupings. In `--dry-run` mode the command exits here. Otherwise the user confirms before proceeding.

### Phase 3: Execute

Files are first written to a temporary staging directory inside the vault. Only after all writes succeed does the staging directory get moved into place - this provides atomic execution and prevents partial migrations from corrupting the vault.

- **Standalone files** are copied into the appropriate tier with generated YAML frontmatter
- **Project folders** (directories with multiple related files) are compacted into a single reference entry using extract-and-attribute. If structured extraction fails, a Table of Contents listing is used as a fallback. Original files are preserved under archive/
- **Session files** (matching date-based or session-* naming patterns) are archived verbatim

### Audit Trail

Every migration writes `MIGRATION-LOG.md` at the vault root. This log maps every source path to its destination, records any compaction decisions, and serves as a "find my file" safety net. The command is idempotent - running it again skips files that were already migrated (matched by source path in the log).

Post-migration, `kb index` and `kb doctor` run automatically to ensure the vault is consistent.

## Future Work

### Homebrew formula

Package as a proper Homebrew tap for `brew install kb` with versioned releases and automatic updates.

### Git hooks for auto-index

Provide installable git hooks that run `kb index` automatically on pre-commit, ensuring INDEX.md never goes stale in version-controlled vaults.

### Template system

Customizable entry templates for common patterns - decision records, runbooks, investigation notes, session summaries. `kb add --template decision "Use Token Bucket for Rate Limiting"` scaffolds the right structure.

### Vault linking

Reference entries across multiple vaults. A project vault can reference entries in a personal vault without copying them. Implemented via relative paths in frontmatter cross-references.
